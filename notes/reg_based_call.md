# Register-Based Calling Convention for IR JIT

## Overview

The WasmEdge IR JIT uses the **dstogov/ir** library to compile Wasm functions
to native x86-64 machine code.  This document describes the register-based
calling convention that replaced the original buffer-based convention for
JIT-to-JIT function calls.

### Before (buffer-based)

Every JIT-compiled function had the uniform signature:

```c
uint64_t func(JitExecEnv *env, uint64_t *args);
```

Arguments were marshalled into a `uint64_t[]` buffer by the caller, and the
callee loaded them back with `ir_LOAD` from buffer offsets.  All calls —
direct, host, and indirect — went through the `jit_direct_or_host` trampoline.

**Problems:** every call required N stores + N loads through memory, defeating
the IR optimizer's ability to propagate values across call boundaries.  The
trampoline added an extra indirect-call layer and prevented inlining.

### After (register-based)

Each JIT-compiled function declares its Wasm parameters as individual
`ir_PARAM` nodes with native IR types.  The IR library assigns SysV ABI
registers automatically.  Direct JIT-to-JIT calls use `ir_CALL_N` with a
typed `ir_proto`, passing arguments directly in registers.

```
ret func(JitExecEnv *env, arg0, arg1, ...)
         ^^^ rdi           ^^^ rsi/xmm0, rdx/xmm1, ...
```

Host calls and `call_indirect` still use buffer-based ABI through their
respective C trampolines, since those cross the JIT/interpreter boundary.

## Architecture

### Call categories

| Call kind | Path | ABI |
|-----------|------|-----|
| Direct call to JIT function (not import, not skipped) | Inline `ir_CALL_N` with typed proto | Register (SysV) |
| Direct call to import or skipped function | `jit_host_call` trampoline | Buffer (`uint64_t[]`) |
| `call_indirect` | `jit_call_indirect` trampoline | Buffer (`uint64_t[]`) |
| Host-to-JIT entry (`IRJitEngine::invoke`) | `irJitInvokeNative` dispatch | Register (SysV) |

### Component diagram

```
                  Host / Interpreter
                         |
                  IRJitEngine::invoke()
                         |
                  irJitInvokeNative()      ← autogenerated dispatch
                         |                    casts & calls with typed args
                   JIT function A
                    /          \
          direct call        host call
          (inline)           (trampoline)
              |                  |
        JIT function B    jit_host_call()
                                 |
                          WasmEdge Executor
```

## Callee Side: Function Prologue

**File:** `ir_builder.cpp`, `WasmToIRBuilder::initialize()`

Each Wasm function declares parameters as consecutive `ir_PARAM` nodes:

```cpp
// Parameter 1: execution environment (always in rdi)
EnvPtr = ir_PARAM(IR_ADDR, "exec_env", 1);

// Parameters 2..N+1: Wasm arguments with native types
for (uint32_t i = 0; i < ParamTypes.size(); ++i) {
    ir_type irType = wasmTypeToIRType(ParamTypes[i]);
    Locals[i] = ir_PARAM(irType, name, i + 2);
}
```

The IR library assigns registers per the SysV x86-64 ABI:
- Integer/pointer args: RDI, RSI, RDX, RCX, R8, R9 (then stack)
- FP args: XMM0–XMM7 (then stack)

Since `exec_env` always consumes RDI (integer slot 0), Wasm integer
parameters start at RSI.  FP parameters get their own XMM counter,
independent of the integer counter.

### Return type

`Ctx.ret_type` is set to the Wasm function's native return type (IR_I32,
IR_I64, IR_FLOAT, IR_DOUBLE, or IR_VOID).  The IR library generates
the appropriate `ret` instruction, placing integer results in RAX and
FP results in XMM0.

## Caller Side: Direct Calls

**File:** `ir_builder.cpp`, `WasmToIRBuilder::visitCall()`, direct-call branch

For a Wasm `call` to a defined (non-import) function:

```cpp
// 1. Load callee pointer from FuncTable
ir_ref FuncPtr = ir_LOAD_A(ir_ADD_A(FuncTablePtr,
    ir_CONST_ADDR(ResolvedFuncIdx * sizeof(void *))));

// 2. Build typed prototype: ret func(JitExecEnv*, arg0, arg1, ...)
ProtoParams[0] = IR_ADDR;  // exec_env
for (i = 0; i < NumArgs; ++i)
    ProtoParams[i + 1] = wasmTypeToIRType(ParamTypes[i]);

ir_ref DirectProto = ir_proto(ctx, IR_FASTCALL_FUNC, DirectRetType,
                              TotalParams, ProtoParams.data());

// 3. Emit the call with arguments in registers
ir_ref CallResult = ir_CALL_N(DirectRetType, TypedFuncPtr,
                               TotalParams, DirectArgs.data());
```

### Return type convention

The caller proto's return type is:
- `IR_FLOAT` / `IR_DOUBLE` for FP returns (XMM0)
- `IR_I64` for integer returns *and* void (RAX; the IR x86 backend asserts
  on `IR_VOID` in the address-fusion pass at O2)

For I32-returning callees, the result is declared as I64 in the proto and
truncated with `ir_TRUNC_I32` after the call.

## Host-to-JIT Entry: `irJitInvokeNative`

**Files:** `ir_jit_reg_invoke.h`, `ir_jit_reg_invoke.cpp`,
`ir_jit_reg_invoke_autogen.cpp` (generated by
`gen_ir_jit_reg_invoke_autogen.py`)

When the WasmEdge runtime calls a JIT function from C++ (via
`IRJitEngine::invoke`), it cannot use `ir_CALL_N` — it must produce a real
C function call with the right types in the right registers.

### Two-level dispatch

1. **`irJitInvokeNative`** (hand-written): checks arity (0–6), dispatches
   to the autogenerated function based on whether the return type is void.
   Aborts for arity > 6.

2. **`irJitInvokeNativeAutogen`** (generated): switches on return type, then
   on arity, then on each parameter type combination.  For each combination,
   it `reinterpret_cast`s the function pointer to the exact C signature and
   calls it.

### Autogeneration

The Python generator (`gen_ir_jit_reg_invoke_autogen.py`) produces all
4^n type combinations for arities 0–6, for each of 4 return types + void.
Total: ~21,844 dispatch paths in ~158K lines.

Each path does three things:

1. **Extract arguments** from the `uint64_t *Slots` buffer with type-aware
   casting:
   - I32: `static_cast<int64_t>(static_cast<int32_t>(uint32_t(Slots[i])))`
   - I64: `Slots[i]` (raw uint64_t)
   - F32: `bitCastF32(Slots[i])` (memcpy to float)
   - F64: `bitCastF64(Slots[i])` (memcpy to double)

2. **Cast the function pointer** to the exact typed signature:
   ```cpp
   reinterpret_cast<uint64_t (*)(JitExecEnv*, float, int64_t, ...)>(Fn)
   ```

3. **Return the result** as `uint64_t` (bit-cast for FP types via memcpy).

The compiler sees a direct call with concrete types and generates the correct
register assignment (integer args in GPRs, float/double in XMM registers).

## JitExecEnv Structure

**File:** `ir_jit_engine.h`

```
Offset  Field               Type           Description
──────  ──────────────────  ────────────   ─────────────────────────────────
 0x00   FuncTable           void**         Pointers to JIT native code (by funcIdx)
 0x08   FuncTableSize       uint32_t       Number of entries in FuncTable
 0x0C   _pad                uint32_t       Alignment padding
 0x10   GlobalBase          void*          Base of Wasm global variables
 0x18   MemoryBase          void*          Base of linear memory (memory 0)
 0x20   HostCallFn          void*          → jit_host_call
 0x28   DirectOrHostFn      void*          → jit_direct_or_host (legacy, unused by inline calls)
 0x30   MemoryGrowFn        void*          → jit_memory_grow
 0x38   MemorySizeFn        void*          → jit_memory_size
 0x40   CallIndirectFn      void*          → jit_call_indirect
 0x48   MemorySizeBytes     uint64_t       Current memory size (for bounds checking)
 0x50   RefResultBuf[2]     uint64_t[2]    Scratch buffer for ref-return helpers
```

`JitExecEnv *` is always the first argument (RDI) to every JIT function.
The function prologue loads needed fields from it (FuncTable, GlobalBase,
MemoryBase, trampoline pointers) into IR temporaries.

## Trampolines

### `jit_host_call(env, funcIdx, args)`

Dispatches calls to import functions (host functions) through the WasmEdge
executor.  Marshals `uint64_t[]` args into `ValVariant` parameters,
calls `Executor::jitCallFunction`, and packs the return value back to
`uint64_t`.

Also handles the fast path for `call_indirect` targets that are JIT-compiled:
when `funcIdx` has bit 31 set (`0x80000000 | tableSlot`), it resolves the
funcref from the table and calls `irJitInvokeNative` directly if the target
is a JIT function.

### `jit_call_indirect(env, tableIdx, elemIdx, typeIdx, args, retTypeCode)`

Delegates to `Executor::jitCallIndirect` which performs the Wasm-specified
type check (matching against the module's type section) before dispatching.
If the target is a JIT function, it uses register-based ABI via
`irJitInvokeNative`; otherwise falls back to the interpreter.

### `jit_memory_grow(env, nPages)` / `jit_memory_size(env)`

Thin wrappers around `MemoryInstance::growPage` and `getPageSize`.
`jit_memory_grow` updates `env->MemoryBase` and `env->MemorySizeBytes`
after a successful grow (the buffer may have been reallocated).

## IR Dump Example

A simple function `f(i32, i32) → i32` that calls another function produces
IR like:

```
l_1 = START(l_N);
uintptr_t d_2 = PARAM(l_1, "exec_env", 1);   // rdi
int32_t d_3 = PARAM(l_1, "p0", 2);            // esi
int32_t d_4 = PARAM(l_1, "p1", 3);            // edx

// Load FuncTable from exec_env
uintptr_t d_5 = ADD(d_2, 0);
uintptr_t d_6, l_6 = LOAD(l_1, d_5);

// Load callee pointer from FuncTable[targetIdx]
uintptr_t d_7 = ADD(d_6, targetIdx * 8);
uintptr_t d_8, l_8 = LOAD(l_6, d_7);

// Direct call with typed proto: int64_t callee(env, p0, p1)
uintptr_t d_9 = PROTO(d_8, func(uintptr_t, int32_t, int32_t): int64_t __fastcall);
int64_t d_10, l_10 = CALL/3(l_8, d_9, d_2, d_3, d_4);

// Truncate I64 result to I32 for Wasm
int32_t d_11 = TRUNC(d_10);
l_N = RETURN(l_10, d_11);
```

The IR backend maps `d_3` to ESI, `d_4` to EDX, and the CALL passes them
through directly — zero stores, zero loads.

## Known Limitations

1. **Arity > 6 for host-to-JIT entry:** `irJitInvokeNative` aborts for
   functions with more than 6 Wasm parameters.  JIT-to-JIT calls have no
   arity limit (the IR backend handles stack spilling).

2. **Single return value:** Only the first Wasm return type is handled.
   Multi-value returns would require struct-return ABI or additional
   convention.

3. **`DirectOrHostFn` in JitExecEnv:** Loaded in the prologue but no longer
   used by inline direct calls.  Retained for ABI stability; can be removed
   in a future cleanup.

4. **Remaining O2 crash:** `rust-compression` crashes at O2 only (passes at
   O0 and O1). Root cause is a wasm memory corruption cascade from an
   O2-compiled function. See `notes/regalloc_bug.md` Remaining Issue #2.
   `shootout-ed25519` was fixed by Bug #2 (prev_use_ref duplicate arg fix).

## Bug Fix: O0 Register Clobbering

See `notes/reg_call_bug.md` for the full write-up of the IR library O0 bug
where `tmp_reg` (assigned to RCX by the simple allocator) clobbered the 4th
register argument during stack-argument setup for calls with >6 args.
Fix: use a dedicated `IR_REG_RAX` scratch for stack args in pass 3 of
`ir_emit_arguments`.
