**Branch:** `ir_jit`

## Root Cause: `__multi3` Called Through Indirect Memory-Marshaled ABI

The ed25519 kernel performs elliptic-curve scalar multiplication over the
Curve25519 field (mod 2^255-19). The C compiler that produced the wasm binary
emits calls to `__multi3` (a compiler-rt 128-bit integer multiply helper) for
every field-element multiply. The wasm module has 17 locally-defined functions
and 3 imports; `__multi3` is `func[19]` (117 bytes of wasm code, IR func 016).

In LLVM AOT, `__multi3` is trivially inlined (well below the inline threshold).
In the IR JIT, **each wasm function is compiled as a separate `ir_ctx`**, so
`__multi3` cannot be inlined. Every call goes through the full indirect
call ABI.

### Per-Call Overhead

The IR builder (`lib/vm/ir_builder.cpp:2885-2926`) compiles every wasm-to-wasm
direct call as an indirect call through a function-pointer table, with all
arguments serialized into a shared memory buffer (`SharedCallArgs`):

```
// Caller side (IR for each __multi3 call):
STORE SharedCallArgs+0x00, dest_ptr    // marshal arg 0 (i32)
STORE SharedCallArgs+0x08, lo1         // marshal arg 1 (i64)
STORE SharedCallArgs+0x10, hi1         // marshal arg 2 (i64)
STORE SharedCallArgs+0x18, lo2         // marshal arg 3 (i64)
STORE SharedCallArgs+0x20, hi2         // marshal arg 4 (i64)
func_ptr = LOAD FuncTablePtr[0x98]     // load __multi3 pointer from table
CALL (env_ptr, SharedCallArgs)         // indirect call, 2 native args

// Callee side (__multi3, IR func 016):
dest  = LOAD args+0x00                 // unmarshal arg 0
lo1   = LOAD args+0x08                 // unmarshal arg 1
hi1   = LOAD args+0x10                 // unmarshal arg 2
lo2   = LOAD args+0x18                 // unmarshal arg 3
hi2   = LOAD args+0x20                 // unmarshal arg 4
membase = LOAD env+0x18                // load wasm memory base
// ... 6 MULs + shifts + adds ...
STORE wasm_mem[dest],   result_lo      // write 128-bit result
STORE wasm_mem[dest+8], result_hi
RETURN
```

**Per-call cost:** 5 stores (marshal) + 1 load (func ptr) + indirect call/ret +
5 loads (unmarshal) + 1 load (mem base) + 2 stores (result) = **14 memory ops +
indirect branch** of pure ABI overhead, *before* any caller-saved register
spill/reload.

### Cascading Effects Beyond Direct Overhead

The indirect call ABI causes secondary performance losses that likely exceed the
direct marshaling cost:

1. **Register pressure across calls.** Each CALL clobbers all caller-saved
  registers (rax, rcx, rdx, rsi, rdi, r8-r11 on x86-64). With 809 __multi3
   calls in `__original_main`, the LSRA must spill and reload live values
   around every call. In the inlined version, these values would stay in
   registers across the multiply.
2. **Lost cross-call optimization.** Adjacent __multi3 calls often share
  operands (e.g., the same field element multiplied by different constants
   19 and 38 for Curve25519 reduction). With inlining, the optimizer can CSE
   shared subexpressions and schedule independent multiplies for ILP. Without
   inlining, each call is an optimization barrier.
3. **Function-pointer reload after each call.** Because the CALL may modify
  arbitrary memory (the compiler must assume this), the function-pointer
   address `d_198 = FuncTablePtr + 0x98` is reloaded via LOAD before *every*
   call. There are 809 identical loads of the same address in
   `__original_main`.

## Secondary Issue: `__multi3` Uses 32-bit Schoolbook Multiply

The `__multi3` implementation in the wasm binary splits each 64-bit operand into
two 32-bit halves and performs 6 x 64-bit multiplies (schoolbook on halves):

```
// From IR func 016 (__multi3):
d_15 = MUL(d_6, d_12)     // lo1 * hi2
d_16 = MUL(d_10, d_8)     // hi1 * lo2
d_20 = MUL(d_19, d_18)    // hi_half(lo1) * hi_half(lo2)
d_24 = MUL(d_23, d_22)    // lo_half(lo1) * lo_half(lo2)
d_26 = MUL(d_22, d_19)    // lo_half(lo2) * hi_half(lo1)
d_31 = MUL(d_23, d_18)    // lo_half(lo1) * hi_half(lo2)
```

On x86-64, `mul r64` produces a full 128-bit result in `rdx:rax`. The entire
`__multi3` could be replaced by 3 native `mul` instructions (lo*lo + cross
terms), cutting the multiply work in half. This is a wasm32 target limitation
(the C compiler doesn't know about 64x64->128 hardware multiply), but the
IR JIT could recognize and replace it.

### Multiply Constants

The ed25519 field reduction multiplies by small constants:


| Constant | Value | Usage                                   |
| -------- | ----- | --------------------------------------- |
| `c_192`  | 19    | Curve25519 prime reduction (2^255 - 19) |
| `c_182`  | 38    | 2 * 19, used in reduction               |
| `c_187`  | 2     | Doubling                                |


These are all small enough for the 3-operand `imul r64, r64, imm` form, which
avoids the RAX/RDX constraint of the 2-operand form. The IR backend does select
`IR_IMUL3` for these (verified in `thirdparty/ir/ir_emit_x86.h:3887-3893`), so
the *inline* multiply-by-constant path is already efficient. The problem is that
these multiplies happen *inside* `__multi3` or between calls, not across call
boundaries where the optimizer could exploit them.

## Why LLVM AOT Doesn't Have This Problem

LLVM compiles the entire wasm module as a single LLVM IR module. `__multi3`
(117 bytes of wasm) is well below LLVM's default inlining threshold. After
inlining:

- Zero memory marshaling; values stay in registers.
- No indirect call overhead.
- LLVM's register allocator (graph-coloring) sees the full live range across
the multiply, avoiding unnecessary spills.
- LLVM can apply LICM, CSE, and instruction scheduling across multiply
boundaries.
- LLVM may further lower __multi3 to use hardware 64x64->128 multiply.

This explains why LLVM JIT (1050ms) matches AOT (1046ms) — both inline __multi3.

---

## Why the IR JIT Cannot Inline `__multi3`

The IR JIT compiles each wasm function as a separate `ir_ctx`
(`lib/vm/ir_jit_engine.cpp:120-187`). The compilation flow:

1. For each wasm function, create a fresh `ir_ctx`.
2. Walk wasm bytecode and emit IR nodes via `WasmToIRBuilder`.
3. Call `ir_jit_compile()` which runs SCCP, GCM, LSRA, etc. on that single
  function's graph.
4. Emit native code and register with GDB JIT interface.
5. Store the native code pointer in a function table.

At step 2, when a `wasm::Call` to `__multi3` is encountered, the builder emits
an indirect call through the function table (`ir_builder.cpp:2886-2919`):

```cpp
ir_ref FuncPtr = ir_LOAD_A(ir_ADD_A(
    ValidFT,
    ir_CONST_ADDR(ResolvedFuncIdx * sizeof(void *))));
// ... marshal args to SharedCallArgs ...
ir_ref CallResult = ir_CALL_N(DirectRetType, TypedFuncPtr, 2, DirectArgs);
```

The callee's IR graph is not available — it may not even be compiled yet. The
IR library's `IR_OPT_INLINE` operates within a single `ir_ctx` and cannot reach
across compilation boundaries.

## Proposed Fixes

### Fix 1: Inline `__multi3` in the IR Builder (highest impact, targeted)

When `visitCall` in `ir_builder.cpp` encounters a call to `__multi3`
(detectable by function index and type signature `(i32, i64, i64, i64, i64) -> void`), emit the multiply operations directly as IR instructions instead of
generating a CALL. The IR builder already has access to the wasm arguments on
the value stack.

**Pseudocode:**

```cpp
case recognized_multi3:
  // Pop args: dest(i32), lo1(i64), hi1(i64), lo2(i64), hi2(i64)
  // Emit: result_lo = lo1 * lo2
  //        cross1 = lo1 * hi2
  //        cross2 = hi1 * lo2
  //        result_hi = cross1 + cross2 + carry_from(result_lo)
  // Store result_lo and result_hi to wasm memory at dest
  break;
```

**Expected impact:** Eliminates ~3200 indirect calls per execution. Each call
site shrinks from ~14 memory ops + indirect branch to ~6 ALU ops + 2 stores.
Rough estimate: 30-50% of the 672ms gap.

### Fix 2: Use x86-64 Hardware 128-bit Multiply (combined with Fix 1)

Replace the 6-multiply schoolbook +ultiply needs only 3 `mul` instructions:

```
mul_lo     = lo1 * lo2        (128-bit: rdx:rax)
result_lo  = rax
carry      = rdx
cross1     = lo1 * hi2        (only need low 64 bits)
cross2     = hi1 * lo2        (only need low 64 bits)
result_hi  = carry + cross1 + cross2
```

This requires adding an IR opcode or intrinsic for widening multiply
(MUL_WIDE or similar) in `thirdparty/ir/`.

### Fix 3: Register-Based Calling Convention (broader impact)

Replace the SharedCallArgs memory buffer with register-based argument passing
for wasm-to-wasm calls. x86-64 has 6 integer argument registers
(rdi/rsi/rdx/rcx/r8/r9 in SysV); most wasm functions have <= 6 args. This
would reduce per-call overhead even when inlining isn't possible.

**Impact:** Benefits all kernels, not just ed25519. Would require changes to
both the call site in `ir_builder.cpp:2885-2926` and the function prologue.

### Fix 4: Module-Level Compilation (long-term)

Compile frequently-called small functions (below a size threshold) into the
caller's `ir_ctx`, allowing the IR library's native inliner (`IR_OPT_INLINE`)
to operate across wasm function boundaries. This is the most general solution
but requires significant refactoring of the compilation pipeline.
